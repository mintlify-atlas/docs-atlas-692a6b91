LiteFarm is structured as a Lerna monorepo. The repository contains three application packages plus an end-to-end test suite, all managed under a single root `package.json` alongside orchestration files for Docker and ngrok. Understanding how the packages relate to each other — and which supporting services they depend on — is the starting point for effective contribution.

## Monorepo layout

```
LiteFarm/
├── packages/
│   ├── api/          # Express.js REST API and job scheduler
│   ├── webapp/       # React progressive web app
│   ├── shared/       # Locales and constants shared across packages
│   └── end-to-end/   # Cypress end-to-end test suite
├── lerna.json        # Lerna workspace configuration
├── docker-compose.yml
└── ngrok/
```

The root `lerna.json` declares `"packages": ["packages/*"]` and manages versioning across the workspace. Each package maintains its own `package.json`, Node version (`.nvmrc`), and dependency lockfile.

## How the packages interact

The webapp communicates with the API over HTTP. In development the API listens on port `5001` (configurable via `PORT` in `packages/api/.env`) and the webapp dev server runs on port `3000`.

```
Browser (localhost:3000)
    │  HTTPS JSON requests
    ▼
packages/api  (localhost:5001)
    │  Knex queries
    ▼
PostgreSQL (localhost:5433 in Docker)
    │  S3-SDK calls
    ▼
MinIO (localhost:9000)
```

The export job scheduler is a separate Node process (`src/jobs/index.js`) that runs alongside the API. It connects to the same Redis instance used by Bull and makes HTTP calls back to the API to fetch certification data.

```
packages/api/src/jobs/index.js
    │  Bull queue (Redis)
    ▼
Redis (localhost:6380 in Docker)
    │  S3 uploads
    ▼
MinIO
```

## Supporting services

Running `docker compose up` from the repo root starts all development dependencies defined in `docker-compose.yml`.

<AccordionGroup>
  <Accordion title="PostgreSQL (db)">
    Image `postgres:13`. The Docker container exposes port `5433` on the host (mapped from the default `5432` inside the container) to avoid conflicts with a locally installed Postgres. Two databases are expected: `pg-litefarm` for development and `test_farm` for the test suite.

    ```yaml title="docker-compose.yml"
    db:
      container_name: litefarm-db
      image: postgres:13
      ports:
        - "5433:5432"
      environment:
        POSTGRES_PASSWORD: "postgres"
    ```
  </Accordion>
  <Accordion title="MinIO (object storage)">
    Drop-in replacement for AWS S3. Used for farm images, uploaded documents, and certification export files. The API references MinIO through the AWS S3 SDK by pointing `MINIO_ENDPOINT` to `http://localhost:9000`. A helper container (`create_minio_bucket`) automatically creates the `minio-dev` bucket on first start.
  </Accordion>
  <Accordion title="Redis + export server">
    Redis 7 and the export server are started only when you include the `export` Docker Compose profile: `docker compose up export`. The export server reads crop locale files from `packages/webapp/public/locales` at startup, then processes Bull jobs that generate organic certification PDF exports.
  </Accordion>
  <Accordion title="Imaginary (image compression)">
    `h2non/imaginary:1.2.4` is a stateless HTTP microservice for image resizing and compression. The API sends image data to Imaginary before uploading to MinIO. In development, set `LOCAL_IMAGINARY=http://localhost:8088` in `packages/api/.env`.
  </Accordion>
</AccordionGroup>

<Note>
  The `redis` and `export` services use Docker Compose profiles. Run `docker compose up export` to include them. The `db`, `minio`, and `imaginary` services start by default with `docker compose up`.
</Note>

## Shared package contents

`packages/shared` is a lightweight package that avoids duplicating content between the API and the webapp.

```
packages/shared/
├── constants/
│   └── error.js     # Shared error code constants
└── locales/
    ├── de/
    ├── es/
    ├── fr/
    ├── hi/
    ├── ml/
    ├── pa/
    └── pt/          # Translation strings for supported languages
```

The webapp's Vite config maps the `@shared` alias to this directory:

```typescript title="packages/webapp/vite.config.ts"
resolve: {
  alias: {
    '@shared': path.resolve(__dirname, '../shared'),
  },
},
```

The export scheduler copies locale files from `packages/webapp/public/locales` into `packages/api/src/jobs/locales` at startup (see the `scheduler` npm script), so crop translation keys are consistent across the certification PDF and the UI.
